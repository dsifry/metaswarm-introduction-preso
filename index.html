Knowledge

Patterns, gotchas, decisions, anti-patterns — grows with every PR

Agents are specialized roles. Skills are workflows that coordinate agents. Commands are how you interact. And the knowledge base is how the system gets smarter over time.
Research → Plan → Plan Review → Design Review → Decompose → Execute → Final Review → PR → Shepherd

Each phase has quality gates — the system won’t let you skip steps.

Every task follows this pipeline. The key is that each phase has quality gates built in. You can't skip from plan to code without design review. You can't merge without passing adversarial review.

Trust Nothing, Verify Everything

Agent reports:

“All tests pass ✓”

×

Orchestrator verifies:

Runs tests independently

✓

The orchestrator never trusts self-reports. It verifies by running tests itself.

This is the core philosophy. When a coding agent says 'tests pass', the orchestrator doesn't take its word for it. It runs the tests itself. It's like having a QA team that independently verifies everything.

It learns.

{ "type": "pattern", "fact": "Always mock external APIs in tests" }
{ "type": "gotcha", "fact": "This API returns 200 even on errors" }
{ "type": "decision", "fact": "Use repository pattern for data access" }

Every PR teaches the system something new. Git-native, version-controlled.

The knowledge base lives in git as JSONL files. Agents prime from it before each task. So that gotcha you hit last week? The agents already know about it.

The Friendliest Install

$ claude
> Read through https://github.com/dsifry/metaswarm and
  install it for my project.
Reading metaswarm documentation… Analyzing your project structure… Scaffolding agents, skills, and commands… ✓ metaswarm configured for your project

Just tell Claude what you want.

This is the key insight. You don't need to memorize commands. Open Claude Code in your project and just tell it to read the metaswarm repo and set it up. It will read the documentation, understand your project structure, and adapt the scaffold to your stack.

What Claude Set Up

your-project/ ├── CLAUDE.md ← project instructions ├── .claude/ │ ├── agents/ ← 18 specialized agents │ ├── skills/ ← 9 workflow skills │ ├── commands/ ← 7 slash commands │ └── rubrics/ ← 6 review rubrics ├── knowledge/ ← git-native knowledge base │ ├── patterns.jsonl │ ├── gotchas.jsonl │ └── decisions.jsonl └── .coverage-thresholds.json
Claude read your project, understood its structure, and set up metaswarm tailored to it. The CLAUDE.md file is the entry point — it tells Claude Code about your project and references all the agents and skills.

Your First Task

Step 1: Create a GitHub issue describing a small feature

Step 2: Run it

/project:start-task <issue-number>

Pick something small — a new endpoint, a UI component, a bug fix.

Pick something small for your first run. A new endpoint, a UI component, a bug fix. Create a GitHub issue for it — just a few sentences describing what you want. Then use the start-task command and watch what happens.

Watch the Workflow

Now watch. The researcher explores your codebase first. The architect creates a structured plan. Reviewers validate it — both the plan review gate and the design review gate. Then the coder implements using TDD with adversarial review. A PR gets created automatically. And the PR shepherd monitors it all the way to merge.
